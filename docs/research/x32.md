# Behringer X32 / Midas M32 — OSC remote control

Research note backing `internal/device/definitions/x32.yaml`. The X32 family is
controlled over **OSC carried in UDP**. The MIDI implementation (Program Change
for scenes, CC for faders/mutes/pans) exists but is a small, coarse subset of
the OSC surface, so this device speaks OSC.

## Transport

- **Protocol:** OSC 1.0 messages over **UDP**.
- **Port:** the console listens on **UDP 10023** and replies to the source port
  the client used. (XR-series / X-Air consoles use 10024; the X32 itself is
  10023.)
- **One console, many clients:** several UDP clients can talk to one X32
  simultaneously. UDP packets can be silently dropped, so heavy `/node` floods
  can overrun buffers — pace requests.

## OSC command structure

```
<address> <typetag> [<args>...]
```

- Address example: `/ch/01/mix/fader`
- Type tag: `,` (none), `,i` (int32), `,f` (float32), `,s` (string), or a
  combination such as `,siss`.
- **Set** = send the address with a value argument:
  `/ch/01/mix/fader ,f 0.5`
- **Get** = send the address with *no* argument; the console echoes the current
  value back: `/ch/01/mix/fader` → `/ch/01/mix/fader ,f 0.5`

## Address families

| Family | Range | Purpose |
| --- | --- | --- |
| `/ch/[01..32]/...` | 01..32 (zero-padded) | input channels |
| `/auxin/[01..08]/...` | 01..08 | aux inputs |
| `/fxrtn/[01..08]/...` | 01..08 | FX returns |
| `/bus/[01..16]/...` | 01..16 | mix buses |
| `/mtx/[01..06]/...` | 01..06 | matrices |
| `/main/st/...` | — | main stereo (LR) |
| `/main/m/...` | — | main mono (M/C) |
| `/dca/[1..8]/...` | 1..8 (single digit) | DCA groups |
| `/config/mute/[1..6]` | 1..6 | mute group masters |
| `/-action/...` | — | actions (scene/snippet recall, card ops, undo) |
| `/-stat/...` | — | transient console state (solo, selected, screen) |
| `/fx/[1..8]/...` | 1..8 | effects slots |

Common per-strip sub-addresses (channels, buses, main):

- `/mix/fader` — float level (see curve below)
- `/mix/on` — int 0/1 (mute / unmute, see semantics below)
- `/mix/pan` — float pan/balance, 0.5 = center
- `/mix/st`, `/mix/mono` — int 0/1 routing to main LR / mono
- `/mix/[01..16]/level` — bus send levels (per channel)
- `/config/name` — string, max 12 chars
- `/config/color` — int 0..15
- `/config/icon` — int (icon index)
- `/config/source` — int routing source index

## Value encodings

### Faders / levels — float 0.0..1.0 over a piecewise dB curve

On the wire a fader/level is a **float 0.0..1.0**. The protocol doc describes
the underlying range as `[-90, +10, 1024]`: a scale from **-90 dB to +10 dB in
1024 steps**. The mapping is **piecewise linear in dB** (not a single linear
relationship). The well-known X32 curve, `f` being the 0..1 wire value:

```
f in [0.5 , 1.0 ]:  dB = f * 40  - 30      # 1.00 -> +10 dB, 0.75 ->  0 dB, 0.50 -> -10 dB
f in [0.25, 0.5 ):  dB = f * 80  - 50      # 0.50 -> -10 dB, 0.25 -> -30 dB
f in [0.0625,0.25): dB = f * 160 - 70      # 0.25 -> -30 dB, 0.0625 -> -60 dB
f in [0.0 , 0.0625):dB = f * 480 - 90      # 0.0625 -> -60 dB, 0.0 -> -90 dB (-inf in practice)
```

Sanity check from the doc: `/ch/01/mix/fader ,f 0.8250` corresponds to **+3 dB**
(`0.825 * 40 - 30 = 3`), and `0.75 -> 0 dB`. So **~0.75 ≈ unity (0 dB)**.

The YAML models faders as `type: float, min: 0.0, max: 1.0, unit: dB`. Callers
that want a specific dB value should convert with the curve above; the wire
value is always the 0..1 float.

### On / mute — int 0/1, ON means *unmuted*

`/.../mix/on` (and `/dca/N/on`) is an int with values `{OFF, ON}` = `{0, 1}`.

**Semantics:** `1` (ON) means the strip is **active / unmuted**; `0` (OFF) means
it is **muted**. This is the opposite of a "mute" boolean — sending `on = 1`
*unmutes*. Modeled as an enum `{ off: 0, on: 1 }`.

Mute group masters at `/config/mute/[1..6]` use the same `{OFF, ON} = {0, 1}`
encoding, where `1` engages (mutes) the group.

### Pan — float 0.0..1.0, center = 0.5

`/.../mix/pan` is a float where `0.0` = hard left, `0.5` = center, `1.0` = hard
right (internally `-100..+100`). The doc's example sets channel 02 to "half
right" with `,f 0.7500`.

### Color — int 0..15

`/.../config/color` is an int `0..15` representing
`OFF, RD, GN, YE, BL, MG, CY, WH` followed by the inverted variants
`OFFi, RDi, GNi, YEi, BLi, MGi, CYi, WHi`.

### Name — string (max 12 chars)

`/.../config/name ,s "My Guitar"`. (The current YAML schema types strings as
`int`; the address and intent are documented here and the field is included so
the control surfaces, even though string payloads are a schema follow-up.)

### Scene / snippet recall — int

- `/-action/goscene ,i N` — recall saved scene `N` (0..99).
- `/-action/gosnippet ,i N` — recall saved snippet `N` (0..99).

## Querying state and subscribing

- **`/info`** — returns server name and firmware/version strings, e.g.
  `/info ,ssss V2.05 osc-server X32 2.12`. Good for discovery / liveness.
- **`/xinfo`** — returns network identity (IP, name, model, firmware).
- **`/<address>`** with no argument — **Get**: console echoes the current value.
- **`/node ,s <path>`** — returns a node's parameters as a single text line
  (the format used in scenes/snippets/presets). Useful to read a whole group at
  once, e.g. `/node ,s ch/01/mix`.
- **`/xremote`** — subscribe to **all** parameter changes. After `/xremote`, the
  console pushes updates for ~10 seconds; the client must re-send `/xremote`
  (or `/renew`) within that window to stay subscribed.
- **`/subscribe` / `/batchsubscribe` / `/formatsubscribe`** — targeted
  subscriptions to specific addresses at a chosen update rate, also renewed via
  `/renew`.
- **`/showdump`** — dumps all scene/snippet data (large; reliable wired link
  recommended).
