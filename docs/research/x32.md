# Behringer X32 — OSC implementation research (verified live)

Research note backing `internal/device/device-types/x32.yaml`. Unlike the other
device notes (which are transcribed from manuals), this one was **verified
against a live console** on the LAN — a **Behringer X32 RACK**, firmware
**4.13**, OSC server **V2.07** — so the address shapes, value encodings and
reply formats below were observed directly, not just read from the reference.

> Installation-specific details (the actual console name, IP, channel/bus
> layout, scenes, routing of one rig) are intentionally **not** in this public
> note. A per-rig snapshot lives in `docs/private/` (gitignored). Everything
> here is generic to the X32 family.

The X32 is controlled over **OSC/UDP on port 10023** (X-Air/XR uses 10024 — a
different protocol surface). There is no official OSC spec; the de-facto
reference is the Patrick-Gilles Maillot document, cross-checked here against the
console.

## Sources

- **Unofficial X32 OSC protocol** (Patrick-Gilles Maillot) — the canonical
  reverse-engineered reference for the whole address tree:
  <https://sites.google.com/site/patrickmaillot/x32>
  (mirror PDF: <https://tostibroeders.nl/wp-content/uploads/2020/02/X32-OSC.pdf>)
- **X32 MIDI table** (for context; the OSC tree is far richer):
  <https://behringer.world/wiki/doku.php?id=x32_midi_table>
- **Live probe** — an X32 RACK on FW 4.13. The address shapes, encodings and
  reply formats in this note were observed directly.

## Console identity (verified)

Three query addresses return identity without arguments:

| Query | Reply args | Example (generic) |
|-------|-----------|-------------------|
| `/info` | `server-fw, server-name, model, console-fw` | `V2.07`, `osc-server`, `X32RACK`, `4.13` |
| `/xinfo` | `ip, name, model, fw` | `<ip>`, `<console-name>`, `X32RACK`, `4.13` |
| `/status` | `state, ip, name` | `active`, `<ip>`, `<console-name>` |

These make `osc.Discover` cheap: a `/info` (or `/xinfo`) probe to a candidate
host/broadcast identifies an X-family console and its model in one round-trip.
`model` distinguishes variants (`X32`, `X32RACK`, `X32C`, `X32PRODUCER`, …) —
useful, but the **OSC tree is identical across the whole X32 family** (32 input
channels, 16 buses, 6 matrices, 8 DCAs, 8 aux in, 8 FX returns), so a single
`x32` definition serves all of them. The rack/compact variants just lack
physical faders/screen; nothing in the OSC surface changes.

## Two ways to read the tree — and their different units

This is the most important protocol nuance, confirmed on the box:

1. **Direct address query** — send the parameter address (e.g.
   `/ch/05/mix/fader`) with no arguments. The console replies with the **raw
   wire value**: float `0.0..1.0` for faders/levels/pan, `int` for on/color/
   source/scene. This is what `osc.Send` writes and what the YAML value specs
   model. Examples observed:

   | Address | Reply |
   |---------|-------|
   | `/ch/05/mix/fader` | float `0.7175` |
   | `/ch/05/mix/pan` | float `0.3` |
   | `/ch/05/eq/1/f` | float `0.265` |
   | `/ch/05/eq/1/g` | float `0.5` (= 0 dB, EQ gain centre) |
   | `/ch/01/dyn/thr` | float `1.0` |
   | `/ch/01/preamp/trim` | float `0.5` |
   | `/ch/01/config/source` | int (input index) |
   | `/ch/01/config/color` | int `2` (= GN) |

2. **`/node` query** — send to address `/node` with a **single string arg**
   being the node path **without the leading slash** (e.g. `ch/01/mix`). The
   console replies with a single string: the node path followed by its values
   rendered in **console-display units** (dB, color *names*, `ON/OFF`,
   `-oo` for −∞). This is the human-readable form and is handy for reconcile/
   dumps, but it is **not** the wire encoding — do not feed it back as a set.

   Observed examples (names blanked):
   - `node 'ch/01/config'` → `/ch/01/config "" 2 GN 1`
     (tuple: **name, icon#, color-name, source#**)
   - `node 'ch/01/mix'` → `/ch/01/mix ON -84.4 OFF +2 OFF   -oo`
     (tuple: **on, fader-dB, lr-assign, pan, mono, mono-level**)
   - `node 'ch/01/dyn'` → `/ch/01/dyn OFF COMP PEAK LOG 0.0 3.0 1 0.00 10 10.0  151 POST 0 100 OFF`
   - `node 'ch/01/preamp'` → `/ch/01/preamp +0.0 OFF OFF 24  20`
   - `node 'dca/1'` → `/dca/1 ON   0.0`

So: **`/node` for reading/dumping, direct addresses for setting.** The dB↔float
fader mapping stated in the design (piecewise, `0.75 ≈ 0 dB`, `0.5 ≈ −10 dB`,
range −90..+10 dB) is consistent with the readings (`-84.4 dB` at fader `0.0117`).

### `config/...` tuple shapes (verified)

- **input channel** `config` = `name icon# color-name source#`
  (e.g. `/ch/01/config "" 2 GN 1`)
- **bus / matrix / main / dca** `config` = `name icon# color-name`
  (no `source` field): e.g. `/bus/01/config "" 11 GN`
- **aux in** `config` = `name icon# color-name source#`
  (e.g. `/auxin/05/config "" 62 RD 37`) — aux-in sources are 33..40.

Colors enumerate `OFF,RD,GN,YE,BL,MG,CY,WH` (0..7) then inverted `*i` (8..15);
the int query returns 0..15 and the `/node` form returns the name (`GN`, `WH`,
`OFF`, …). The YAML already models color as `int 0..15` — correct.

## Aggregate / list addresses (verified)

Some addresses return one packed reply for many elements:

- `/config/chlink` → **16** `ON/OFF` flags = the stereo-link state of channel
  **pairs** 1·2, 3·4, … 31·32 (not 32 values).
- `/config/mute` → **6** `ON/OFF` flags = mute-group masters 1..6.
- `/-stat/solosw` → **80** flags = the full solo bus.
- `/config/solo`, `/config/routing/*`, `/fx/<n>/par`, `/-prefs`, `/-action`,
  `/-stat` each return a fixed packed record.

`/config/routing` returns the active routing-preset name, and the IN/OUT blocks
resolve sources in groups of eight, e.g.:
- `/config/routing/IN` → `UIN1-8 UIN9-16 UIN17-24 UIN25-32 AUX1-4`
- `/config/routing/OUT` → `OUT1-4 OUT5-8 OUT9-12 OUT13-16`

The `UINx` blocks mean channels are fed from the USB/recording card rather than
local mic pres; `INx` blocks mean local/AES50 inputs. Which blocks are active is
per-installation (see the private rig note).

## Subscriptions / inbound (for `osc.Listen`)

Not all state is push by default. For the design's reconcile/MIDI-learn path the
reference (and standard X32 practice) is:

- **`/xremote`** (no args) — sent at least every ~10 s; the console then mirrors
  **every parameter change** back to the sender's address/port. This is the
  cheap way to keep desired-state reconciled with hand-tweaks on the console.
- **`/subscribe <addr> <rate>`** + periodic **`/renew`** — for high-rate data
  like meters (`/meters/<n>`), returned as OSC **blobs**.

`osc.Listen` should bind the same UDP socket it sends from (the X32 replies to
the **source** port), parse the reply (address → typetag → args), and emit
`transport.Event`s. A probe parsing `i/f/s/T/F/b` type tags is sufficient.
`/node`-style reads are display-unit strings, so reconcile should prefer
direct-address reads (wire values) where exactness matters.

## Notable address families (verified to exist)

- **Main**: `/main/st` (LR) and `/main/m` (mono) both exist, each with its own
  `config` + `mix` subtree.
- **FX**: `/fx/[1..8]/type` is the engine type as a string (e.g. `VRM`);
  `/fx/[1..8]/par` returns the full parameter array for that engine.
- **Head amps**: `/headamp/NNN` (000-based) = `gain-dB, phantom`
  (e.g. `+8.0 OFF`).
- **Scenes/snippets**: `/-show/showfile/scene/NNN` holds saved scenes;
  `/-action/goscene` / `/-action/gosnippet` recall by number. A console with no
  saved scenes returns empty scene slots, and `goscene` then has nothing to
  recall — scenes must be created on the console (or via OSC) first.

## Implications for the project

- **`x32.yaml` is correct** as far as it goes: the addresses, the float
  `0.0..1.0` fader/level encoding, `mix/on` as int `0/1`, pan float `0.5`
  centre, color int `0..15`, and `goscene`/`gosnippet` all match the live box.
  Now also modeled (verified to exist): the `config` **`source`** field for
  input/aux channels, plus **`/main/m`** and **`/main/st/config`**.
- **`osc.Discover`** → `/info`/`/xinfo` probe (returns model + fw in one packet).
- **`osc.Connect`/`Send`** → plain UDP to `host:10023`; encode address +
  `,`-typetag + args (`i/f/s`). No pairing (already a no-op in the interface).
- **`osc.Listen`** → reuse the send socket, periodic **`/xremote`** for change
  mirroring, `/subscribe`+`/renew` for meters; parse replies into
  `transport.Event`.

same way — is kept in `docs/private/x32-rig.md`, which is gitignored.
